\begin{quote}
Per bcrypt implementation, only the first 72 bytes of a string are used. Any extra bytes are ignored when matching passwords. Note that this is not the first 72 {\itshape characters}. It is possible for a string to contain less than 72 characters, while taking up more than 72 bytes (e.\+g. a UTF-\/8 encoded string containing emojis). \end{quote}


As should be the case with any security tool, anyone using this library should scrutinise it. If you find or suspect an issue with the code, please bring it to the maintainers\textquotesingle{} attention. We will spend some time ensuring that this library is as secure as possible.

Here is a list of BCrypt-\/related security issues/concerns that have come up over the years.


\begin{DoxyItemize}
\item An \href{http://www.openwall.com/lists/oss-security/2011/06/20/2}{\texttt{ issue with passwords}} was found with a version of the Blowfish algorithm developed for John the Ripper. This is not present in the Open\+BSD version and is thus not a problem for this module. HT \href{https://twitter.com/zooko}{\texttt{ zooko}}.
\item Versions {\ttfamily \texorpdfstring{$<$}{<} 5.\+0.\+0} suffer from bcrypt wrap-\/around bug and {\itshape will truncate passwords \texorpdfstring{$>$}{>}= 255 characters leading to severely weakened passwords}. Please upgrade at earliest. See \href{https://github.com/kelektiv/node.bcrypt.js/wiki/Security-Issues-and-Concerns\#bcrypt-wrap-around-bug-medium-severity}{\texttt{ this wiki page}} for more details.
\item Versions {\ttfamily \texorpdfstring{$<$}{<} 5.\+0.\+0} {\itshape do not handle NUL characters inside passwords properly leading to all subsequent characters being dropped and thus resulting in severely weakened passwords}. Please upgrade at earliest. See \href{https://github.com/kelektiv/node.bcrypt.js/wiki/Security-Issues-and-Concerns\#improper-nul-handling-medium-severity}{\texttt{ this wiki page}} for more details.
\end{DoxyItemize}\hypertarget{README.md_autotoc_md325}{}\doxysubsection{\texorpdfstring{Compatibility Note}{Compatibility Note}}\label{README.md_autotoc_md325}

\hypertarget{README.md_autotoc_md341}{}\doxysubsection{\texorpdfstring{A Note on Timing Attacks}{A Note on Timing Attacks}}\label{README.md_autotoc_md341}
Because it\textquotesingle{}s come up multiple times in this project and other bcrypt projects, it needs to be said. The {\ttfamily bcrypt} library is not susceptible to timing attacks. From codahale/bcrypt-\/ruby\#42\+:

\begin{quote}
One of the desired properties of a cryptographic hash function is preimage attack resistance, which means there is no shortcut for generating a message which, when hashed, produces a specific digest. \end{quote}


A great thread on this, in much more detail can be found @ codahale/bcrypt-\/ruby\#43

If you\textquotesingle{}re unfamiliar with timing attacks and want to learn more you can find a great writeup @ \href{https://codahale.com/a-lesson-in-timing-attacks/}{\texttt{ A Lesson In Timing Attacks}}

However, timing attacks are real. And the comparison function is {\itshape not} time safe. That means that it may exit the function early in the comparison process. Timing attacks happen because of the above. We don\textquotesingle{}t need to be careful that an attacker will learn anything, and our comparison function provides a comparison of hashes. It is a utility to the overall purpose of the library. If you end up using it for something else, we cannot guarantee the security of the comparator.
